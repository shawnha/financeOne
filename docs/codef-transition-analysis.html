03 전환 전 vs 전환 후 — 시스템 구조

현재 상태 (As-Is)

Gowid를 FinanceOne + ExpenseOne 공용
  • ExpenseOne과 1:1 매칭 트리비얼 (같은 소스)
  • 전환 비용 제로
  • Gowid SPOF — 장애 시 양쪽 모두 정지
  • 은행(Codef) / 카드(Gowid) 백엔드 이원화
  • 해외결제 누락 가능성 (4월 Vercel/Claude 일시 미반영)
  • 취소 이력 소실 (net만 남음)
  • 교차 검증 불가
추천

전환 후 (To-Be)

FinanceOne = Codef / ExpenseOne = Gowid (분리)
  • 은행·카드 모두 Codef로 백엔드 통일
  • 교차 검증 — 두 독립 소스로 결측/오류 조기 발견
  • 이중화 SPOF 제거
  • 취소 별도 row 보존 (감사·추적 완결)
  • 해외결제 실시간 반영
  • 백그라운드 cron이 자동 인수인계
  • ExpenseOne ↔ Codef 매칭에서 시차 >1일 건은 수동 매칭 증가

05 리스크 & 완화

심각도 리스크 완화 방안
High Step 1 DELETE 실패 시 Gowid 원본 손실 삭제 전 SELECT * ... → /tmp/lotte_backup_2026-04.csv 백업 + 삭제 row ID 기록
Mid ExpenseOne 매칭 8건이 일시 깨짐 transaction_expenseone_match는 expense_id unique — DELETE CASCADE로 자동 해제. Codef sync 후 같은 card_last4+date+amount로 자동 재매칭. 실패 건은 /expenseone-match 메뉴에서 수동 매칭.
Mid Codef 롯데가 Gowid와 다른 시차로 거래를 기록해 재무제표 차이 Dry-run 결과 Codef net vs Gowid fresh 차이 0.5%. 회계법인 ±5% 허용 한계 내. 월말에 한 번 더 검증.
Low Codef 롯데 연결 재차단 (CF-12803 재발) 오늘 연결 성공 확인 완료. 재발 시 codef_api_log에 transactionId 자동 기록 → Codef 지원 문의 즉시 가능.
Low 취소건 UI 노출로 거래내역 혼잡 후속 작업 D (거래내역 "취소건 숨기기" 토글) — 기본 ON으로 두면 일상 뷰 깔끔.
