컬럼 매핑

Excel 컬럼인덱스→ ParsedTransaction 필드처리 방식
거래일시 (B)col 1datedatetime이면 .date(), 문자열이면 공백 앞부분만 파싱 ("2026.01.02 09:15" → 2026-01-02)
적요 (C)col 2description 앞부분"체크우리"이면 is_check_card = True
기재내용 (D)col 3counterparty거래처명으로 사용
지급 (E)col 4amount + type="out"값이 있고 >0이면 지출
입금 (F)col 5amount + type="in"값이 있고 >0이면 수입
잔액 (G)col 6사용하지 않음파싱하지 않음 — 검증에 활용 가능하지만 현재 무시

counterparty = row[3]  # 기재내용

체크카드 처리

적요가 "체크우리"이면 is_check_card = True 설정. 이 거래는 우리카드 체크계좌 거래와 중복이므로 마킹됨.

알려진 문제 1: 잔액 컬럼 무시 잔액(G) 컬럼을 파싱하지 않아서 기말잔고 검증이 불가능합니다. 잔액 컬럼의 마지막 행 값을 balance_snapshots에 자동 저장하면 수동 입력 없이 기초/기말 잔고를 설정할 수 있습니다.

description 생성

description = f"롯데카드 {counterparty}"  # 예: "롯데카드 카카오T"
알려진 문제 2: 취소 건 처리 취소여부가 "Y"인 행은 완전히 건너뜁니다. 하지만 취소된 거래 자체는 원래 승인 건과 별개 행입니다. 취소 건을 건너뛰면 원래 승인 금액은 그대로 남아서 지출이 과대 계상될 수 있습니다.

예: 1/15 카카오T ₩16,800 (승인) → 1/16 카카오T ₩16,800 (취소 Y) → 현재 로직은 취소만 SKIP하고 승인은 그대로 INSERT.

올바른 처리: 취소 건이 있으면 원래 승인 건도 함께 제거하거나, 취소를 type="in" (환불)으로 INSERT해야 합니다.

업로드 시 중복 감지 문제 있음

파일: backend/routers/upload.py (line 64-73)

현재 로직

// 중복 판단 기준: (entity_id, date, amount, counterparty)
SELECT id FROM transactions
WHERE entity_id = %s AND date = %s AND amount = %s AND counterparty = %s
LIMIT 1

// 중복이면 is_duplicate = True 마킹
// ⚠️ 하지만 INSERT는 무조건 실행됨 → 같은 파일 2번 업로드하면 2배로 들어감
핵심 문제 4: 중복이어도 INSERT됨 is_dup = True로 마킹만 하고 INSERT는 항상 실행합니다. 같은 파일을 2번 업로드하면 모든 거래가 2배로 들어갑니다.

수정 필요: 중복이면 INSERT를 건너뛰어야 합니다.
핵심 문제 5: 중복 판단 키가 부족 (date, amount, counterparty)만으로 판단하면:
❌ 같은 날 같은 거래처에 같은 금액을 2번 보낸 경우 → 두 번째 거래가 중복으로 처리됨
❌ 리앤목특허법인에 1/14 660,000원을 2번 이체한 경우 → 1건만 남음

수정 필요: (date, amount, counterparty, description) 또는 (date, amount, counterparty, source_type, file_id)를 키로 사용해야 합니다.
문제 6: 체크카드 무조건 중복 마킹 is_check_card = True이면 DB 조회 없이 무조건 is_dup = True. 하지만 은행 거래 업로드 전에 카드를 먼저 업로드하면 은행에 대응 건이 없어도 중복 처리됨.

올바른 처리: 체크카드 중복은 은행 거래가 존재할 때만 마킹해야 합니다.

문제 요약 + 수정 필요 사항

#문제심각도현재 동작수정 방향
1 우리은행 잔액 컬럼 무시 MEDIUM 잔액(G) 컬럼 파싱 안 함 마지막 행 잔액을 balance_snapshots에 자동 저장
2 카드 취소 건 → 원래 승인 잔존 HIGH 취소 행만 SKIP, 원래 승인은 그대로 취소 건을 type="in" (환불)로 INSERT하거나, 대응 승인 건 찾아서 함께 제거
3 같은 파일 재업로드 시 2배 CRITICAL 중복 마킹만 하고 INSERT는 실행 중복이면 INSERT 건너뛰기 (if is_dup: continue)
4 같은 날 같은 금액 정상 거래 누락 HIGH (date, amount, counterparty)만으로 중복 판단 description도 키에 포함, 또는 file_id 기반 중복 방지
5 체크카드 무조건 중복 MEDIUM is_check_card이면 무조건 is_dup 은행 거래 존재 여부 확인 후 마킹
6 우리카드 연도 누락 가능 LOW 메타에서 연도 추출 실패 시 현재 연도 파일명에서 확실하게 추출하는 로직 보완
