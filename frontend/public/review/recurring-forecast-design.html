is_recurring 설계 논의

"고정"으로 체크했는데 왜 forecast에 안 나오지?

1. 현재 어떻게 동작하는가

내부계정 관리에서 "고정" 체크 │ ▼ internal_accounts.is_recurring = TRUE │ │ ✗ forecast에 자동 반영 없음 │ ▼ 예상 탭에서 수동으로 항목 추가 ← 여기서 빠뜨리면 영원히 누락 │ ▼ forecasts 테이블에 행 생성 (is_recurring = TRUE) │ ▼ 다음 달: "전월 반복 가져오기" 클릭 │ ▼ forecasts.is_recurring = TRUE인 행만 복사 │ │ ✗ internal_accounts.is_recurring은 무시 ▼ 새 달 forecast 생성
문제: is_recurring이 두 곳에 존재합니다. 내부계정(internal_accounts)과 예상(forecasts) 각각에 있고, 서로 연결되어 있지 않습니다.
내부계정에서 "고정"을 체크해도 forecast에는 아무 일도 일어나지 않습니다.

2. 기대 vs 현실

사용자 기대

"고정 = 매달 자동으로 forecast에 포함"

임차료를 "고정"으로 체크하면,
4월 예상을 열었을 때 임차료가
이미 들어가 있어야 한다.

실제 동작

"고정 = 전월 복사 시 필터 조건일 뿐"

"전월 반복 가져오기" 버튼을 눌러야 하고,
전월 forecast에 없으면 복사 자체가 안 됨.
최초 1회 수동 등록이 반드시 필요.

결과: 22개 반복 계정 중 11개가 forecast에 빠져 있음. 이 11개의 3월 실제 지출 합계: 약 3,100만원

3. 이것은 설계 결함인가?

설계 결함이라고 볼 수 있는 이유

  • 기대 불일치: "고정"이라는 UI 라벨은 "매달 고정적으로 나간다"를 의미. 하지만 실제로는 forecast 생성과 무관
  • 이중 상태: 같은 개념(반복 여부)이 두 테이블에 따로 존재하며 동기화 안 됨
  • 빠뜨리기 쉬움: 새 반복 계정 추가 → 내부계정에서 "고정" 체크 → 예상에 수동 등록을 깜빡함 → 해당 항목이 영원히 forecast에서 누락
  • 실제 피해: 3월 forecast 정확도가 24%(카드)인 주요 원인

하지만 현재 방식이 의도적일 수 있는 이유

  • 금액 문제: "고정"은 "반복 발생"이지, "금액이 매달 같다"는 아님. AI SaaS 구독은 매달 다름 (사용량에 따라). 자동 생성하려면 금액을 어디서 가져올지 정해야 함
  • 검토 강제: 매월 수동으로 확인하고 금액을 조정하는 것이 더 정확한 forecast를 만들 수 있음
  • 오염 방지: 실적 없는 계정이 자동으로 forecast에 들어가면 noise가 됨 (예: "패키징 외주"가 3개월째 0원인데 매달 forecast에 나옴)

4. 핵심 트레이드오프

자동 생성 수동 관리 (현재) 장점: 장점: • 빠뜨림 없음 • 금액을 매달 검토 • 새 반복 계정 추가 시 즉시 반영 • 불필요한 항목 없음 • is_recurring의 의미가 명확 • 사용자가 완전한 통제 단점: 단점: • 금액을 어디서 가져올까? • 11개 항목 누락 (실제 발생) • 0원인 달에도 항목이 보임 • forecast 정확도 24% (카드) • "이 항목 왜 있지?" 피로감 • is_recurring의 존재 이유 불명확

5. 세 가지 접근법

A. 자동 생성 + 검토 UI 추천

매월 초 forecast 열 때, is_recurring=TRUE인데 해당 월 forecast에 없는 계정을 자동 감지 → "추가 제안" 배너로 표시 → 사용자가 한 번에 확인/추가/무시 선택

장점
  • 빠뜨림 방지
  • 금액을 사용자가 직접 확인
  • 강제 삽입 아님 — 제안일 뿐
  • is_recurring의 의미가 명확해짐
단점
  • UI 추가 필요 (제안 배너)
  • 제안 금액 기준 필요 (전월 실적? 평균?)
동작 방식:
예상 탭 열기 → "반복 항목 중 3개가 빠져있습니다" 배너 → [전부 추가] [하나씩 확인] [무시]
금액 기준: 최근 3개월 평균 (실적 없으면 0원으로 표시 + 입력 요청)

B. 완전 자동 생성 (Silent)

is_recurring=TRUE 계정은 매월 forecast에 자동 삽입. 금액은 전월 실적 또는 최근 3개월 평균. 사용자가 삭제할 수 있음.

  • 가장 간단 — 빠뜨림 완전 제거
  • "전월 반복 가져오기" 불필요
  • 실적 없는 항목도 들어감 (noise)
  • 금액이 자동이라 검증 안 하고 넘길 수 있음
  • "이 항목 왜 있지?" 피로감

C. 현재 유지 + 경고 추가

현재 수동 방식 유지. 단, is_recurring=TRUE인데 forecast에 없는 항목이 있으면 경고 표시.

  • 변경 최소 — 경고 UI만 추가
  • 사용자 통제 유지
  • 여전히 수동 등록 필요
  • 경고를 무시하면 동일 문제 반복
  • is_recurring의 이중 상태 미해결

6. 더 깊은 질문: is_recurring이 두 곳에 필요한가?

현재 이중 상태의 근본 원인:

위치역할필요한가?
internal_accounts.is_recurring "이 계정은 매달 발생하는 항목이다" 필요 — 계정의 성격을 정의
forecasts.is_recurring "이 forecast 행을 다음 달로 복사할까" 불필요할 수 있음 — internal_accounts에서 파생 가능

만약 Option A를 선택하면, forecasts.is_recurring은 더 이상 필요 없습니다.
"전월 반복 가져오기"도 internal_accounts.is_recurring을 직접 참조하면 됩니다.
단일 진실 소스: internal_accounts.is_recurring

Option A 적용 후: 내부계정에서 "고정" 체크 │ ▼ internal_accounts.is_recurring = TRUE ← 단일 진실 소스 │ ▼ 예상 탭 열기 │ ▼ 자동 감지: is_recurring=TRUE인데 forecast 없는 항목 │ ▼ "반복 항목 N개가 빠져있습니다" 배너 표시 │ ├── [전부 추가] → 최근 3개월 평균으로 자동 생성 ├── [하나씩 확인] → 개별 확인 UI └── [무시] → 이번 달은 스킵

결론

현재 구조는 설계 결함입니다. "고정"이라는 라벨이 약속하는 것과 시스템이 하는 것이 다릅니다.

추천: Option A — 자동 감지 + 제안 배너. 강제 삽입은 아니지만, 빠뜨린 항목을 확실히 잡아줍니다.
forecasts.is_recurring은 폐기하고, internal_accounts.is_recurring을 단일 진실 소스로 사용합니다.
