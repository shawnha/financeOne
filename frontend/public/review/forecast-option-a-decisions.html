1.type ('in'/'out') 추론 전략

문제: internal_accounts 테이블에 type 컬럼이 없는데 forecasts UNIQUE constraint 는 (entity_id, year, month, internal_account_id, type) 에 포함. → missing-recurring API 가 type 없이는 forecast 행을 만들 수 없음. 어떻게 추론할지.
추천: 1A — DRY + minimal diff. 기존 transactions 데이터만으로 충분. 1B 는 블로킹, 1C 는 이번 범위 초과.

2.배너 Dismissal 저장 위치

문제: 사용자가 "이번 달 무시" 눌렀을 때 어디에 저장할지. 브라우저별 분리 vs 서버 통일.
추천: 2A — 사용자 규모(1-2명) 대비 오버엔지니어링 방지. 향후 필요 시 settings 로 마이그레이션 쉬움.

3.suggested_amount 0원 케이스 처리

문제: internal_account 가 is_recurring=TRUE 지만 최근 3개월 거래 0건 → 제안 금액 0원. [전부 추가] 눌렀을 때 어떻게 처리할지.
추천: 3A — 사용자 가시성 최대화. 0원 행은 수정 핸들. 데이터 누락 방지.

4.기존 "전월 반복 가져오기" 버튼 공존

문제: 배너 도입 후 기존 버튼과 기능 겹침. 사용자 혼란 가능. (forecast-tab.tsx:1681-1705)
추천: 4A — 기능 목적이 다름. 배너는 "누락 fix", 버튼은 "금액 refresh". 분리된 채로 둠.

5.suggested_amount 계산 공식

문제: 최근 3개월 평균을 어떻게 계산할지. 같은 internal_account에 in/out 섞여 있을 수 있음 (환불).
추천: 5B — 대표성·실제성 최고. 5A는 환불 왜곡, 5C는 과대추정 (사용자 취향).

6.테스트 커버리지 — E2E 포함 여부

문제: 배너 + 2개 API 의 테스트 범위. pytest unit 만 vs Playwright E2E 추가.
