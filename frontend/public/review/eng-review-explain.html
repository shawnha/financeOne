P2 FIXED .find() 선형 탐색 → Map 기반 접근

문제: 마지막 실제일의 예상 잔고를 찾을 때 .find()로 배열을 처음부터 순회합니다. 31일이라 느리진 않지만, 만약 배열에 해당 day가 없으면 null이 되어 조정 예상 전체가 원래 예상으로 fallback됩니다. 즉, 조정 기능이 조용히 꺼집니다.

BEFORE (위험)

// 배열 순회 — day가 없으면 null
const lastForecastAtActualDay =
  schedule.points.find(
    p => p.day === lastActualDay
  )?.balance ?? null

// worst-case도 같은 문제
worstCasePoints.find(...)

AFTER (안전)

// Map으로 O(1) 접근 + 명시적
const forecastBalanceByDay = new Map(
  schedule.points.map(p =>
    [p.day, p.balance])
)

forecastBalanceByDay.get(lastActualDay)
// worst-case도 동일하게 Map
데이터 접근 비교: .find() 방식: [day0, day1, day2, ... , day14?, ... day31] → → → → → 순회 14번 (없으면 null → 조정 꺼짐!) Map 방식: { 0: 161M, 1: 160M, ..., 14: 130M, ..., 31: 107M } O(1) 직접 접근 (없으면 null → 명시적 fallback)

인덱스 vs Day 기반 접근: schedule.points: [day0, day1, day2, ..., day31] ← day0 포함 (32개) worst_case_points: [day0, day1, day2, ..., day31] ← day0 포함 (32개) 인덱스 기반: points[5] = day5 ✓ (지금은 맞음) 만약 worst_case에 day0이 빠지면? worst[5] = day6 ← 밀림! Map 기반: forecastByDay.get(5) = 항상 day5 worstByDay.get(5) = 항상 day5

수정 요약

  • .find() → Map: 선형 탐색을 O(1) 접근으로. sparse data에서 조정 기능이 조용히 꺼지는 위험 제거.
  • 인덱스[i] → Map.get(day): day 0 추가/제거 시 인덱스 어긋남 방지. day 기준으로 항상 정확.
  • 코드 일관성: 조정 예상과 worst-case가 동일한 Map 패턴 사용.
